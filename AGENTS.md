# AGENTS.md

Правила для AI-правок в `fixed_solvers`.

## 1) Базовые ограничения

- Язык: C++.
- Среда: Windows.
- Не менять кодировку файлов.
- Не использовать подавление неиспользуемых параметров через `(void) parameter`.
- Не использовать `using namespace std;`.
- Не объявлять вложенные классы или структуры.
- При разработке соблюдать SRP и Rule-of-7: не перегружать типы множеством ответственностей и не оставлять перегруженные методы.

## 2) Структура проекта и include

- Использовать централизованные агрегирующие заголовки (`fixed\fixed.h`).
- При добавлении/удалении файлов обновлять `CMakeLists.txt`.
- При переименовании файлов/директорий использовать `git mv`.

## 3) Нейминг и namespace

- Snake_case для переменных, функций, `enum` и значений `enum`.
- Типы (структуры/классы): snake_case + суффикс `_t`.
- Имена тестов `TEST(Suite, Case)` — в CamelCase, без транслита; `Suite` должен отражать тестируемую подсистему (SUT), `Case` — проверяемое поведение и начинаться с глагола (`Returns/Throws/Creates/Should...`).
- Имена должны быть информативными и однозначно отражать смысл сущностей и действий.
- Не использовать `using namespace std;`.
- Для `namespace` использовать принятый формат:
  `namespace ... {` и следующей строкой `;`.

## 4) Стиль кода и API

- Перед возвратом сложного значения вводить промежуточную переменную (например, `result`) и возвращать ее отдельной строкой.
- Использовать alias-типы вместо дублирования длинных шаблонов.
- Следовать правилу `Pass only what you need` (передавать только нужные контейнеры/поля).
- Соблюдать DRY: повторяющиеся блоки выносить в helper/общие функции.
- Не оставлять лишние пустые строки между полями и методами в `struct`/`class`.

## 5) Комментарии и документация в коде

- В `.h` для функций/структур/классов писать Doxygen (`///`, `@brief`, при необходимости `@param/@return`).
- Не дублировать Doxygen-комментарии в `.cpp`, если они уже есть в `.h`.
- Для тестов `TEST(...)` Doxygen-комментарий должен по-русски формулировать проверяемое поведение, согласованное с именем теста.
- Не использовать Doxygen-комментарии внутри тел функций/методов.
- Внутри функций использовать только короткие `//` для нетривиальных мест.

## 6) Тесты

- Не оборачивать `TEST(...)` в `namespace`.
- Требование: при добавлении или изменении любого теста обязательно оформлять явные секции `Arrange`, `Act`, `Assert` в указанном порядке. Отсутствие хотя бы одной секции (`Arrange`, `Act` или `Assert`) считается нарушением правил оформления тестов.
- Внутри тестов явно выделять блоки `Arrange / Act / Assert` и добавлять к каждому блоку краткое пояснение выполняемого шага.
- Формат секций должен быть явным и полным в одной строке комментария, например: `// Arrange: ...`, `// Act: ...`, `// Assert: ...`.
- Никогда не объединять `Act` и `Assert` в одном комментарии: оформлять их только отдельными строками.
- Повторяющиеся длинные тестовые данные выносить в helper-функции/фикстуры.

## 7) Математика и сложность

- Для π использовать `M_PI` из `<cmath>` (без хардкода и локальных `#define`).
- Следить за SRP и общей сложностью: не перегружать классы/методы, крупные логические блоки выносить в отдельные функции.

## 8) Актуальность документации

- При изменении поведения кода обновлять `documents/*` по фактической реализации.
